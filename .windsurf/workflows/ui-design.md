# UI Design — `/ui-design <app-or-screen>`

Use this workflow before any slice that introduces or significantly changes a UI surface (Manager Panel app, Tablet app, modal, etc.).

> **Note**: full UI canon (palette tokens, Bento grid system, motion canon, sound moments, component catalog) lives in `docs/04_UI_PLAYBOOK.md` (to be created in a dedicated session). This workflow is a stub until that playbook exists, and will be expanded then.

## Inputs

- `<app-or-screen>` — the surface being designed (e.g. `MarketApp`, `PlotsTablet`, `OnboardingModal`).

## Steps

1. **Identify the slice** that owns this UI work. Confirm the slice mini-brief exists.

2. **Read** `docs/00_BIBLE.md` §1.1 (palette) + §15 (UI paradigm) + `docs/04_UI_PLAYBOOK.md` if it exists.

3. **Compile design brief** into `docs/slices/<SXX>_<name>.ui_brief.md`:

   ```markdown
   # UI Brief — <SXX> · <app-or-screen>

   ## Context
   <2-3 lines: what this surface is for, who uses it, what they accomplish>

   ## Surface type
   - [ ] Manager Panel (Laptop, fullscreen, dense data)
   - [ ] Tablet field (overlay, glanceable, action-oriented)
   - [ ] Modal / Drawer / Sheet
   - [ ] Inline component

   ## Data shown
   <bullet list of every piece of data on screen, with source: which event/service/DB table>

   ## Actions available
   <bullet list of every action the user can take, with target: which event/service is invoked>

   ## Layout intent
   <Bento? Single column? Split? Reference a known pattern from inspirations>

   ## Components needed
   <list of shadcn primitives + Farm Sonar custom components: GlassCard, BentoGrid, BatchChip, etc.>

   ## Motion intent
   <which moments deserve micro-interaction: card entrance, hover, success notif, error shake>

   ## Inspirations
   <2-3 references: Linear app X, Stripe page Y, Climate FieldView dashboard Z>

   ## Out of scope
   <what this surface explicitly does NOT do (push to next slice or different surface)>
   ```

4. **Hand off to v0.dev**. The founder copies the brief into a v0.dev session. v0.dev produces a mockup. Founder validates visually.

5. **Review checklist** before approving the v0.dev mockup for implementation:
   - [ ] Uses Farm Sonar palette tokens (no random hex).
   - [ ] Uses Geist Sans + JetBrains Mono only.
   - [ ] Uses shadcn primitives where applicable (no reinvented buttons/inputs).
   - [ ] Bento grid where multiple data clusters coexist.
   - [ ] Glassmorphism on cards (`rgba(255,255,255,0.7)` backdrop blur).
   - [ ] Lime neón `#CCFF00` used sparingly, only on key actions / focal data.
   - [ ] Quality letters S/A/B/C/D color-coded per Bible §1.1.
   - [ ] Mono font for IDs (batch_id, IBAN, dates).
   - [ ] No string is hardcoded — all reference i18n keys.
   - [ ] Loading + empty + error states designed.
   - [ ] Mobile/Tablet aspect handled (if surface is in Tablet mode).

6. **Implementation handoff**: once mockup approved, the slice's Frontend Agent prompt (generated by `/spawn-pm`) references the mockup + this brief.

7. **Report to user**: brief path + v0.dev next-step.

## Notes

- Never approve a mockup that fails review checklist items 1-9. Items 10-11 can be deferred to a polish pass within the same slice.
- The UI Playbook (when written) will replace much of this workflow's content with canonical references.
