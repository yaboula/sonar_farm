# Spawn PM — `/spawn-pm`

Use this workflow when a slice is complex enough to require coordinated work across multiple specialized sub-agents (Backend / Frontend / Integration / QA).

## Role definition

When this workflow is invoked, you (Cascade) **adopt the PM Agent role**. You:

- Do **NOT** write production code in this session.
- Do **NOT** make architectural decisions unilaterally — those belong to the founder + Bible.
- DO split the slice work into clear, independent prompts for sub-agents.
- DO produce a single artifact: `docs/slices/<SXX>_<name>.prompts.md` containing one prompt block per sub-agent.

## Inputs

- Active slice mini-brief at `docs/slices/<SXX>_<name>.md`.
- Roadmap entry for the slice at `docs/01_ROADMAP.md`.

## Sub-agent specialties (fixed catalog)

| Agent | Owns |
|---|---|
| **Backend Agent** | Lua server-side logic, DB migrations, services, FSMs, `sonar:farm:*` events, server-authoritative validations. |
| **Frontend Agent** | React 18 + TS strict + Tailwind v4 + shadcn + Framer Motion, NUI shell, Bento apps, Recharts visualizations. |
| **Integration Agent** | `Bridge.*` glue, `ox_inventory`/`ox_target`/`ox_lib` integration, client↔server messaging, anims, props, vehicles, `fxmanifest.lua`. |
| **QA Agent** | Automated tests (server-side Lua tests, Vitest for NUI), smoke test scripts, balance pass, DoD verification. |

## Steps

1. **Read** the active slice mini-brief + the Roadmap entry.

2. **Decide which sub-agents are needed.** Aim for 2-4. Document the reasoning in 2-3 lines.

3. **Decompose the work.** For each chosen sub-agent, identify:
   - Specific deliverables they own (subset of the slice's deliverables).
   - Inputs they need (which DB tables, which events, which existing services).
   - Outputs they produce (files, tables, events, UI).
   - Interface contracts with other sub-agents (events emitted/consumed, props expected, shared types).

4. **Write `docs/slices/<SXX>_<name>.prompts.md`** with the following structure (in English):

   ```markdown
   # Sub-agent prompts for <SXX> — <name>

   > Generated by PM Agent on <date>.
   > Founder: open one fresh Cascade session per sub-agent below, paste the prompt verbatim.

   ## Coordination contract

   <2-3 lines: which events flow between agents, shared types, blocking order if any>

   ## Prompt — Backend Agent

   <full prompt: context references, deliverables, DoD, contracts with other agents>

   ## Prompt — Frontend Agent

   <full prompt>

   ## Prompt — Integration Agent

   <full prompt>

   ## Prompt — QA Agent

   <full prompt>
   ```

5. **Each prompt block must include:**
   - **Context refs**: explicit lines telling the agent to read Bible + Roadmap + this slice's mini-brief + relevant rules.
   - **Scope**: 3-5 bullet points of what they own.
   - **Deliverables checklist**: files to create/edit, with paths.
   - **Interface contracts**: events emitted/consumed, types shared, data flow.
   - **DoD specific to this sub-agent**: how the founder verifies their part is done.
   - **Out of scope**: explicit list of what they must NOT touch (belongs to other agents).

6. **Report to the user** a one-paragraph summary listing:
   - Sub-agents spawned + 1-line role each.
   - Suggested order (parallel vs sequential).
   - Path to the prompts file.

## Notes

- The PM session itself does not implement; the founder will copy each prompt into a fresh session per sub-agent.
- Sequential order is required when one sub-agent's output is input for another (e.g. Backend defines events first, Frontend consumes them).
- Parallel order is fine when sub-agents work on independent files.
- If a slice would need a 5th sub-agent (e.g. Game Designer for anims/MLO), explicitly document why and ask the founder to approve before spawning beyond 4.
