## Coordination contract

S5 has three useful agents: Backend, Integration and QA. Backend owns the migration, server-authoritative `PlotService`, seed sync semantics and event payloads. Integration owns `fxmanifest.lua` load order, config loading, admin command wiring and runtime boot orchestration. QA owns DB shape verification, idempotency checks, lint, smoke documentation and DoD audit.

Suggested order:

1. Backend Agent implements the core domain first: migration + config + service API.
2. Integration Agent wires boot/admin/load-order once Backend has stable file names and service function names.
3. QA Agent can prepare checks immediately, then runs final validation after Backend + Integration are merged.

S5 is complexity M, so these prompts are optional accelerators. If one Cascade session implements the whole slice, use the same contracts below as the implementation checklist.

## Shared S5 contract

### Canonical justification

- Roadmap S5: create `sonar_farm_plots`, delta-calc-ready timestamps, persistent `soil_score`, plot types `extensive`, `horticultural`, `greenhouse`, seed 8 predefined MLO plots, admin reload command, `sonar:farm:plot:state_changed` event.
- Bible §9.2: server-authoritative absolute; DB is source of truth.
- Bible §9.3: table/event/resource naming conventions.
- Bible §9.4: no direct resource calls, no client trust, no hardcoding, no unsafe migrations.
- Bible §13.2: real-time persistence, downtime-safe delta-calc, idempotent boot behavior.
- Bible §17: plot is a cultivable terrain unit with type and persistent soil score.

### Non-negotiable boundaries

- Do not implement planting, wheat lifecycle, crop stages, 3D rendering, item creation, quality formulas, storage, sales or NUI Plots app.
- Do not trust client data for plot state.
- Do not add direct calls to unrelated resources.
- Do not put tunable plot data in service code; use `config/plots.lua`.
- Do not hardcode user-facing messages; use `locales/en.json` and `locales/es.json`.
- Do not change product canon or Bible unless explicitly instructed.

### Baseline schema contract

The final migration may refine types/indexes, but it must preserve these concepts:

| Column                             | Required purpose                                   |
| ---------------------------------- | -------------------------------------------------- |
| `plot_id`                          | Stable config-defined identifier.                  |
| `plot_type`                        | One of `extensive`, `horticultural`, `greenhouse`. |
| `display_name_key`                 | Locale key/stable future UI label reference.       |
| `state`                            | Initial S5 state, expected `fallow`.               |
| `soil_score`                       | Persistent 0-100 integer.                          |
| `coords_x`, `coords_y`, `coords_z` | MLO anchor coordinates.                            |
| `radius`                           | Plot interaction/rendering area for later slices.  |
| `last_updated_ts`                  | UNIX timestamp for delta-calc.                     |
| `planted_ts`                       | Nullable UNIX timestamp for S6.                    |
| `next_stage_ts`                    | Nullable UNIX timestamp for S6 scheduler.          |
| `created_at`, `updated_at`         | DB audit timestamps.                               |

Required migration properties:

- Filename: `sonar_farm_core/database/migrations/004_plots.sql`.
- Table: `sonar_farm_plots` only, plus migration metadata handled by S2 runner.
- Idempotent where possible with `CREATE TABLE IF NOT EXISTS`.
- Rollback procedure documented in `sonar_farm_core/database/README.md` if the migration adds DB state.
- Register in `fxmanifest.lua` with `sonar_farm_migration 'database/migrations/004_plots.sql'`.

### Baseline service contract

Expose exactly one service table following project conventions:

```lua
Sonar.Farm.PlotService = PlotService
```

Expected public methods unless a better minimal API is documented in the mini-brief:

- `Boot() -> boolean`
- `SyncSeededPlots() -> result`
- `ListPlots() -> plots`
- `GetPlot(plot_id) -> plot | nil`
- `CreatePlot(plot) -> ok, row_or_error`
- `UpdatePlotState(plot_id, state_patch, reason) -> ok, row_or_error`
- `ReloadSeededPlots(src) -> result`

All persisted state changes must publish:

```lua
TriggerEvent('sonar:farm:plot:state_changed', payload)
```

Suggested payload shape:

```lua
{
    plot_id = plot_id,
    previous_state = previous_state,
    next_state = next_state,
    reason = reason,
    changed_at = os.time(),
}
```

Do not emit the event for no-op idempotent seed checks unless a row is actually inserted or changed.

## Recommended service semantics

- `Boot()` calls `SyncSeededPlots()` and returns `true` only if seed sync succeeds.
- `SyncSeededPlots()` creates missing configured plots, updates safe config-owned fields only if needed, and does not reset gameplay-owned fields such as `state`, `soil_score`, `planted_ts` or `next_stage_ts` unless explicitly documented.
- `ReloadSeededPlots(src)` uses the same sync path as boot and returns counts for created/updated/unchanged/skipped.
- `UpdatePlotState()` accepts only explicit server-side patch fields needed by current/future slices and stamps `last_updated_ts`.
- `ListPlots()` and `GetPlot()` return normalized Lua tables with predictable keys.
