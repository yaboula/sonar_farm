## Coordination contract

S2 has no frontend work. Backend defines the `Sonar.Farm.DB` wrapper and migration runner behavior. Integration wires the runtime into `fxmanifest.lua`, resource boot, locales and `oxmysql` compatibility. QA validates idempotency, manual rollback, fail-fast behavior and DoD. Work order should be: **Backend first**, then **Integration**, then **QA**. Integration may run in parallel only after Backend has proposed file names and public API.

Shared public contract expected by all agents:

- `sonar_farm_core/server/db/db.lua` exposes a table under `Sonar.Farm.DB` or a clearly documented equivalent.
- `sonar_farm_core/server/db/migrator.lua` exposes a table under `Sonar.Farm.Migrator` or a clearly documented equivalent.
- Migration files live in `sonar_farm_core/database/migrations/NNN_<description>.sql`.
- The applied migration table is `sonar_farm_migrations`.
- All DB code is server-side only.
- No gameplay/domain tables in S2 except the temporary smoke table from Roadmap.

## Your scope

You own the server-side DB foundation logic:

- Implement a minimal `oxmysql` wrapper in `sonar_farm_core/server/db/db.lua`.
- Implement migration discovery/order/applied-state/execution in `sonar_farm_core/server/db/migrator.lua`.
- Create SQL migrations:
  - `sonar_farm_core/database/migrations/001_init_migrations_table.sql`
  - `sonar_farm_core/database/migrations/002_smoke_table.sql`
- Keep the implementation minimal and future-proof without adding gameplay/domain schemas.
- Ensure DB failures produce clear errors and do not pretend the resource is healthy.

## Required behavior

The migrator must:

- Discover migration files in numeric order using a FiveM-compatible resource file pattern.
- Parse the migration ID from the filename prefix: `001`, `002`, etc.
- Record applied migrations in `sonar_farm_migrations`.
- Skip already-applied migrations.
- Apply pending migrations exactly once.
- Log which migrations ran.
- Fail fast if the DB is unreachable or if any migration fails.
- Be idempotent on a second resource restart.

The DB wrapper must expose helpers equivalent to:

- `db.scalar(query, params)`
- `db.row(query, params)`
- `db.rows(query, params)`
- `db.execute(query, params)`
- `db.transaction(queries)`

If exact naming differs, document the reason in the mini-brief before implementation is considered complete.
