## Coordination contract

**Why three slices, one bundle.** S13/S14/S15 all activate quality factor stubs that were
neutral (default score) since B1. They share `sonar_farm_quality_tracking` as the single
persistence target, the same `track()`/`get()` interface already coded in B1, and the same
`ox_target → anim → server callback → factor update` client pattern. Backend implements all
three factor tracker upgrades in one pass; Integration wires three parallel client files;
QA validates the full quality decision surface in a single smoke.

**Shared source of truth.** Every sub-agent reads `docs/slices/B3_factor_activation.md` §8
before writing anything. §8 defines: factor tracker API extensions, all 5 events, PestService
FSM, migration `011_pest_severity.sql`, item config shape, scheduler integration point,
lib.callback registrations, and pest visual approach.

**Hard rules for all sub-agents:**
- No `MySQL.Sync.*` — only `Sonar.Farm.DB.*` (async await).
- No framework-specific calls — bridge only for money/player operations.
- No hardcoded numbers — all thresholds, probabilities, durations in config.
- No hardcoded strings — everything through `locale()`.
- Anti-patterns 1–10 in `.windsurf/rules/05_anti_patterns.md` are absolute.
- Factor tracker methods are **additive** — do NOT replace existing `track()`/`get()` from B1.
- Next migration number is **011** (`011_pest_severity.sql`). Do not use 010.

## Deliverables

### 1. `client/irrigation_interactions.lua`

On resource start:
- Register `ox_target` sphere zone on each plot coords (read from `Config.Farm.Plots` or via
  callback `sonar:farm:storage:list_units` pattern) with option:
  - **"Water crop"** (`fas fa-droplet`): `canInteract` = player has `sonar_water_tank` with
    `charges > 0` AND plot state is `planted/germinating/growing/maturing` (active crop).
  - On trigger: `lib.progressBar({ duration=10000, label=locale('irrigation.interaction.watering'), canCancel=true })`.
  - If not cancelled: `lib.callback.await('sonar:farm:plot:water', plot_id, 1)`.
    Server returns `{ ok, new_score, error }`. On ok: update tank charges metadata client-side.
    On error: notify error.

- Register `ox_target` sphere zone on each well coord from `Config.Farm.Irrigation.well_coords`:
  - **"Refill tank"** (`fas fa-fill-drip`): `canInteract` = player has `sonar_water_tank` with
    `charges < max_charges`.
  - On trigger: `lib.progressBar({ duration=Config.Farm.Irrigation.recharge_duration_ms, ... })`.
  - On complete: notify `locale('irrigation.interaction.refilled')`.
    Client updates tank metadata: `charges = Config.Farm.Irrigation.tank_max_charges`.
    (No server callback needed for refill — tank state is cosmetic; server validates charges on water action.)

- Listen for `sonar:farm:plot:watered` (routed to client via NUI relay if needed, or direct
  `AddEventHandler`). Show brief ox_lib progress indicator resolved.

### 2. `client/fertilization_interactions.lua`

On resource start:
- Register `ox_target` per active plot with option:
  - **"Fertilize"** (`fas fa-seedling`): `canInteract` = player has any fertilizer item in
    inventory. List from `Config.Farm.Items.tools` keys starting with `sonar_fertilizer_`.
  - On trigger: if player has multiple fertilizer types, show `lib.alertDialog` / `lib.inputDialog`
    to select which fertilizer to use (simple select — list items the player actually has).
    Alternatively use `lib.registerContext` menu — whichever is cleaner in ox_lib.
  - `lib.progressBar({ duration=8000, label=locale('fertilization.interaction.applying'), canCancel=true })`.
  - `lib.callback.await('sonar:farm:plot:fertilize', plot_id, selected_item_name)`.
  - On ok: notify score feedback. Item removal is handled server-side.

### 3. `client/pest_interactions.lua`

On resource start:
- Register `ox_target` per active plot with option:
  - **"Treat pest"** (`fas fa-bug-slash`): `canInteract` = `PestFactor.GetStatus` shows active
    pest AND player has `sonar_pesticide_contact` or `sonar_pesticide_systemic`.
    Use `lib.callback.await('sonar:farm:pest:get_status', plot_id)` to check server state.
  - `lib.progressBar({ duration=12000, label=locale('pest.interaction.treating'), canCancel=true })`.
  - `lib.callback.await('sonar:farm:plot:treat_pest', plot_id, pesticide_item_name)`.

- Pest particles map `{ [plot_id] = particle_handle }` at module level.
- `AddEventHandler('sonar:farm:pest:appeared', function(payload) ... end)`:
  - Start looping particle at plot coords (use `StartParticleFxLoopedAtCoord`; dict = `core`,
    effect choose from catalog or use `ent_anim_twisted_vapours_peds` if available — document
    whichever is picked). Store handle in map.
- `AddEventHandler('sonar:farm:pest:treated', function(payload) ... end)`:
  - Stop + remove particle for `payload.plot_id`. Remove from map.
- On resource start: call `lib.callback.await('sonar:farm:pest:get_active')` and spawn particles
  for any already-active pests (server reconnect scenario).
- On `onResourceStop`: stop all active particles.
