## Coordination contract

**Why three slices, one bundle.** Roadmap says S6+S7+S8 close together; their DoDs are
mutually blocking. Backend must finish first because it defines the DB schema, service APIs, and
event payloads that Integration and Frontend consume. Integration and Frontend can start in
parallel once Backend ships the §Shared contracts section of the mini-brief. QA runs last.

**Shared source of truth.** Every sub-agent reads `docs/slices/B1_wheat_lifecycle_item_quality.md`
§8 before writing a single line. §8 defines: DB schemas for `sonar_farm_crops`,
`sonar_farm_batches`, `sonar_farm_quality_tracking`; event payload shapes; `BatchMetadata` TS
interface; wheat stage → prop mapping (hashes from the Asset Catalogue).

**Hard rules for all sub-agents:**
- No `MySQL.Sync.*` anywhere. Only `Sonar.Farm.DB.*` (async, `*.await` pattern).
- No `exports['other_resource']:Method()`. Events or bridge only.
- No hardcoded numbers — everything in `config/crops/wheat.lua` or `config.lua`.
- No hardcoded strings — everything in `locales/en.json` + `locales/es.json`.
- Anti-patterns 1–10 in `.windsurf/rules/05_anti_patterns.md` are absolute.

### 3. `server/lifecycle/crop_lifecycle_service.lua`

Expose `Sonar.Farm.CropLifecycle` with:
- `Plant(plot_id, player_cid, crop_type) -> ok, error` — validate plot exists + is `fallow`,
  insert `sonar_farm_crops` row, insert `sonar_farm_quality_tracking` row (soil_score from plot),
  call `Sonar.Farm.PlotService.UpdatePlotState(plot_id, { state='planted', planted_ts=now, next_stage_ts=... }, 'plant')`,
  emit `sonar:farm:plot:planted`.
- `Harvest(plot_id, player_cid) -> ok, batch_id, error` — validate plot `harvest_ready` + crop row
  exists, calculate quality via `Sonar.Farm.Quality.Calculate(plot_id, harvest_timing_score)`,
  call `Sonar.Farm.PhysicalItem.CreateBatch(params)`, remove seed from `sonar_farm_quality_tracking`,
  update `sonar_farm_crops` to archived state (or delete row — decide in §Architecture notes),
  call `PlotService.UpdatePlotState` back to `fallow` with cooldown `next_stage_ts`,
  emit `sonar:farm:plot:harvested` + `sonar:farm:batch:created`.
- `GetActiveCrop(plot_id) -> crop | nil` — single row query.
- `AdvanceStage(plot_id) -> ok, new_stage, error` — increment stage, recalculate `next_stage_ts`,
  call `PlotService.UpdatePlotState` to sync `state` + `next_stage_ts`, emit `sonar:farm:plot:stage_advanced`.
- `Boot() -> boolean` — logs active crop count, starts scheduler (calls `Sonar.Farm.Lifecycle.Scheduler.Start()`).

All DB ops use `Sonar.Farm.DB` helpers. No `MySQL.Sync`. Use `MySQL.transaction.await` for
multi-table writes (plant writes to 3 tables; harvest writes to 3 tables).

### 4. `server/lifecycle/scheduler.lua`

`Sonar.Farm.Lifecycle.Scheduler` with:
- `Start()` — creates a `CreateThread` loop; waits `Config.Farm.Scheduler.TickSeconds * 1000` ms;
  queries `sonar_farm_crops WHERE next_stage_ts <= UNIX_TIMESTAMP() AND stage < 4`; calls
  `CropLifecycleService.AdvanceStage(plot_id)` for each result. Log tick activity only when
  `Config.Farm.Debug = true`.
- `Stop()` — graceful stop flag checked inside loop.

Anti-pattern §4: **never** `Wait(0)`. Default tick = 30 s.

### 5. `server/items/physical_item.lua`

`Sonar.Farm.PhysicalItem` with:
- `CreateBatch(params) -> batch_id, metadata, error` — generates `batch_id` using §8.5 strategy
  (`sf-` + 8 random hex), assembles full metadata table, inserts into `sonar_farm_batches`,
  calls `ox_inventory` (via event bus, not direct export — see anti-pattern §1) to add item to
  player inventory with metadata, emits `sonar:farm:item:created`.
- `ValidateMetadata(meta) -> ok, error` — validates all required fields present and typed correctly.
  Rejects missing `batch_id`, negative `weight_g`, invalid quality grade, `freshness` out of 0-100.

Note on ox_inventory: adding items server-side uses `exports.ox_inventory:AddItem(source, name, count, metadata)`.
This is a direct export — but it is an **external dependency call to ox_inventory**, which is a
declared dependency, not a cross-resource call between our own resources. This is allowed by
anti-pattern §1 (which forbids calls between Farm Sonar resources, not to declared `dependencies`).
Document this decision clearly in a code comment.
